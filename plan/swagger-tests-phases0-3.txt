IssueFlow - Current Swagger Manual Test Guide
File purpose: document only tests that can be executed RIGHT NOW from implemented code.
Last updated: current codebase state (controllers under /auth, /users, /projects, /tickets, /tickets/{ticketId}/comments)

====================================================================
1) Environment setup
====================================================================

1.1 Start database (PostgreSQL)
- Option A (recommended from project): use compose.yml
  Command:
    docker compose up -d db

- Option B: local PostgreSQL service
  Must match application-local.yaml:
    host: localhost
    port: 5432
    database: issueflow
    username: issueflow
    password: issueflow

1.2 Start Spring Boot app
- From project root:
    ./mvnw spring-boot:run -Dspring-boot.run.profiles=local

1.3 Confirm app is up
- Open Swagger UI:
    http://localhost:8080/swagger-ui/index.html

====================================================================
2) Important precondition for login
====================================================================

Current code has NO automatic seeded user.
If login fails because no user exists, create one local bootstrap user in DB.

2.1 Bootstrap local user (development only)
Run in PostgreSQL query tool:

  CREATE EXTENSION IF NOT EXISTS pgcrypto;

  INSERT INTO users (username, email, full_name, role, password_hash)
  VALUES (
    'swagger_admin_local',
    'swagger_admin_local@example.com',
    'Swagger Local Admin',
    'ADMIN',
    crypt('Password123!', gen_salt('bf'))
  )
  ON CONFLICT (username) DO NOTHING;

2.2 Verify bootstrap user exists

  SELECT id, username, role
  FROM users
  WHERE username = 'swagger_admin_local';

Save returned id as:
  <ADMIN_ID>

====================================================================
3) Swagger auth usage (JWT)
====================================================================

3.1 Login to obtain token
- Endpoint: POST /auth/login
- After success, copy:
  <ACCESS_TOKEN> = response.accessToken

3.2 Authorize in Swagger
- Click "Authorize" (top right lock button).
- Enter token in the JWT field.
  If Swagger expects raw token: paste <ACCESS_TOKEN>.
  If Swagger expects full value: paste Bearer <ACCESS_TOKEN>.

3.3 Common auth requirement
- Only POST /auth/login is public.
- All other business endpoints require JWT.

6) Authentication flow tests
====================================================================

Test A1 - Login with valid credentials
- Purpose: get JWT token.
- Method: POST
- Path: /auth/login
- Authorization required: No
- Headers: Content-Type: application/json
- Request body:
  {
    "username": "swagger_admin_local",
    "password": "Password123!"
  }
- Expected status: 200
- Expected success response:
  {
    "accessToken": "<jwt>",
    "tokenType": "Bearer",
    "expiresIn": 3600
  }
- Verify in response:
  - accessToken is non-empty string
  - tokenType = "Bearer"
  - expiresIn > 0
- Copy:
  <ACCESS_TOKEN>
- DB verify:
  - no new row required for login itself.

Test A2 - Login with invalid username
- Purpose: negative auth check.
- Method/Path: POST /auth/login
- Request body:
  {
    "username": "not_exists_user",
    "password": "Password123!"
  }
- Expected status: 401
- Verify: authentication error (bad credentials/unauthorized).

Test A3 - Login with invalid password
- Purpose: negative auth check.
- Method/Path: POST /auth/login
- Request body:
  {
    "username": "swagger_admin_local",
    "password": "WrongPassword!"
  }
- Expected status: 401

Test A4 - Login with missing required field
- Purpose: validation check.
- Method/Path: POST /auth/login
- Request body:
  {
    "username": "swagger_admin_local",
    "password": ""
  }
- Expected status: 400
- Verify: validation message indicates password is required/not blank.

Test A5 - Get current user after login
- Purpose: verify token-based identity endpoint.
- Method: GET
- Path: /auth/me
- Authorization required: Yes
- Required headers:
  Authorization: Bearer <ACCESS_TOKEN> (via Authorize button)
- Request body: none
- Expected status: 200
- Verify in response:
  - id equals bootstrap/admin user id
  - username/email/fullName/role are present

Test A6 - Access /auth/me without token
- Purpose: protected endpoint check.
- Method/Path: GET /auth/me
- Authorization required: intentionally NOT provided
- Expected status: 401

14) Database verification checklist
====================================================================

Use SQL during/after tests:

Users:
  SELECT id, username, email, role FROM users ORDER BY id DESC;

Projects:
  SELECT id, name, owner_id, deleted_at FROM projects ORDER BY id DESC;

Tickets:
  SELECT id, project_id, status, priority, due_date, is_overdue, deleted_at
  FROM tickets ORDER BY id DESC;

Comments:
  SELECT id, ticket_id, author_id, content, created_at FROM comments ORDER BY id DESC;

Comment mentions:
  SELECT comment_id, mentioned_user_id, created_at
  FROM comment_mentions ORDER BY created_at DESC;

Audit logs (state-changing actions):
  SELECT id, action, entity_type, entity_id, actor_type, actor_user_id, created_at
  FROM audit_logs ORDER BY id DESC;

Revoked tokens:
  SELECT id, jti, expires_at, revoked_at FROM revoked_tokens ORDER BY id DESC;

What to verify:
- create/update/delete actions create expected DB changes
- soft deletes set deleted_at (projects, tickets)
- logout inserts revoke record
- comment mention rows are synchronized with updated content
