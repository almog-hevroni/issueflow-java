========================================================================
1) Purpose of Phase 5 tests
========================================================================

Phase 5 in this codebase is focused on automated/background behavior for tickets.

What "Scheduler + System Automation" means in the current implementation:

1. Automatic overdue escalation scheduler
   - A background scheduler runs periodically.
   - It scans overdue active tickets and escalates priority by one level:
     LOW -> MEDIUM -> HIGH -> CRITICAL.
   - If a ticket is already CRITICAL and still overdue, it sets is_overdue=true.
   - It does not change status.

2. Reset behavior after manual priority update
   - If user manually updates ticket priority via PATCH /tickets/{ticketId},
     is_overdue is reset to false immediately.

3. Automatic/system audit logging
   - Scheduler writes audit logs with:
     action = AUTO_ESCALATE
     actor_type = SYSTEM
   - Existing auto-assignment behavior (implemented in TicketService.createTicket)
     writes:
     action = AUTO_ASSIGN
     actor_type = SYSTEM

Important implementation facts from current code:
- Scheduler exists: TicketEscalationScheduler
- Scheduling is enabled: @EnableScheduling in IssueFlowApplication
- No dedicated API endpoint exists to trigger scheduler directly from Swagger.
  Scheduler runs in background by configured interval only.

4) Scheduler behavior tests (full flow)
========================================================================

Because there is NO manual trigger endpoint for scheduler in current code,
these tests are semi-manual:
1) prepare data through Swagger
2) wait for scheduler interval
3) verify by Swagger GET and SQL

Configured scheduler interval:
- issueflow.ticket.escalation.fixed-delay-ms = 60000 (60 seconds)

Recommended wait:
- 70-90 seconds after setup to ensure at least one cycle executed.

Common SQL verification query:
  SELECT id, title, status, priority, due_date, is_overdue, assignee_id, deleted_at
  FROM tickets
  WHERE id IN (
    <TICKET_LOW_ID>, <TICKET_MEDIUM_ID>, <TICKET_HIGH_ID>, <TICKET_CRITICAL_ID>,
    <TICKET_FUTURE_ID>, <TICKET_NULL_DUE_ID>, <TICKET_DONE_ID>, <TICKET_DELETED_ID>
  )
  ORDER BY id;

Common audit verification query:
  SELECT id, action, entity_type, entity_id, actor_type, actor_user_id, details_json, created_at
  FROM audit_logs
  WHERE entity_type='TICKET'
    AND entity_id IN (<TICKET_LOW_ID>, <TICKET_MEDIUM_ID>, <TICKET_HIGH_ID>, <TICKET_CRITICAL_ID>)
  ORDER BY id DESC;

5) Auto-escalation tests for overdue tickets
========================================================================

Test AE1 - LOW overdue escalates to MEDIUM
- Purpose: verify first escalation step.
- Precondition: <TICKET_LOW_ID> exists with priority LOW, dueDate past, not DONE, not deleted.
- Setup endpoint: POST /tickets (see section 3.5 body).
- Immediate expected: response priority LOW, isOverdue false.
- Wait: 70-90 seconds.
- Verify via Swagger:

  Expected: priority MEDIUM, isOverdue false.
- Verify via SQL:
  SELECT priority, is_overdue FROM tickets WHERE id=<TICKET_LOW_ID>;
  Expected: MEDIUM, false.
- Audit expectation:
  one AUTO_ESCALATE row, actor_type=SYSTEM, entity_id=<TICKET_LOW_ID>.

Test AE2 - MEDIUM overdue escalates to HIGH
- Precondition: <TICKET_MEDIUM_ID> created.
- Wait: 70-90 seconds.
- Verify: GET /tickets/<TICKET_MEDIUM_ID>
  Expected: priority HIGH, isOverdue false.
- SQL expected: HIGH, false.
- Audit expected: AUTO_ESCALATE, SYSTEM.

Test AE3 - HIGH overdue escalates to CRITICAL
- Precondition: <TICKET_HIGH_ID> created.
- Wait: 70-90 seconds.
- Verify: GET /tickets/<TICKET_HIGH_ID>
  Expected: priority CRITICAL, isOverdue false (first CRITICAL step only).
- SQL expected: CRITICAL, false.
- Audit expected: AUTO_ESCALATE, SYSTEM.

Test AE4 - CRITICAL overdue stays CRITICAL
- Precondition: <TICKET_CRITICAL_ID> created as CRITICAL past due.
- Wait: 70-90 seconds.
- Verify: GET /tickets/<TICKET_CRITICAL_ID>
  Expected: priority remains CRITICAL.
- SQL expected: priority CRITICAL.
- Audit: AUTO_ESCALATE may still be created for overdue flag change (see AE5).

- Precondition: CRITICAL + past due + isOverdue currently false.
- Wait: 70-90 seconds.
- Verify:
  GET /tickets/<TICKET_CRITICAL_ID> => isOverdue true.
- SQL:
  SELECT priority, is_overdue FROM tickets WHERE id=<TICKET_CRITICAL_ID>;
  Expected: CRITICAL, true.
- Audit:
  AUTO_ESCALATE with details_json showing overdueBefore false, overdueAfter true.

Test AE6 - Future dueDate ticket not changed
- Precondition: <TICKET_FUTURE_ID> created with dueDate in future.
- Wait: 70-90 seconds.
- Verify:

- SQL expected unchanged row.
- Audit: no AUTO_ESCALATE expected for this ticket.

Test AE7 - Null dueDate ticket not changed
- Precondition: <TICKET_NULL_DUE_ID> created without dueDate.
- Wait: 70-90 seconds.
- Verify unchanged via GET.
- SQL expected unchanged.
- Audit: no AUTO_ESCALATE expected.

Test AE8 - DONE ticket not escalated
- Precondition: <TICKET_DONE_ID> created with status DONE and past due date.
- Wait: 70-90 seconds.
- Verify:
  GET /tickets/<TICKET_DONE_ID>
  Expected unchanged priority, isOverdue stays false.
- SQL expected unchanged.
- Audit: no AUTO_ESCALATE expected.

Test AE9 - Soft-deleted ticket not escalated

Test AE10 - Already-overdue behavior across multiple cycles
- Purpose: show one-step-per-cycle behavior.
- Setup: create HIGH past-due ticket.
- Wait cycle #1 (70-90s): expect CRITICAL + isOverdue false.
- Wait cycle #2 (70-90s): expect CRITICAL + isOverdue true.
- Wait cycle #3 (70-90s): expect unchanged CRITICAL + true (idempotent).
- Verify with GET and SQL after each cycle.
- Audit:
  cycle #1 likely AUTO_ESCALATE (priority change),
  cycle #2 likely AUTO_ESCALATE (overdue flag change),
  cycle #3 no new AUTO_ESCALATE for unchanged state.

- Setup:

- SQL workload check:
  SELECT assignee_id, count(*)
  FROM tickets
  WHERE project_id=<PROJECT_ID>
    AND deleted_at IS NULL
    AND status <> 'DONE'
  GROUP BY assignee_id
  ORDER BY count(*) ASC;

Test AA3 - DONE/deleted excluded from workload counting
- Setup:
  mark some tickets DONE and soft-delete others.
- Create ticket without assigneeId.
- Expected:
  only active non-DONE tickets affect workload.
- Reason from code:
  countOpenAssignedTickets filters deleted_at is null and status <> DONE.

Test AA4 - No DEVELOPER exists
- Setup:
  create project where only ADMIN users exist (or delete dev users from test data).
- Create ticket without assigneeId.
- Expected response:
  assigneeId = null, status 200.
- SQL verify assignee_id is null.

Test AA5 - Equal workload tie-break
- Current implemented behavior:
  developers are loaded by created_at ascending; stream.min keeps first encountered
  on equal workload.
- Setup equal loads for DEV1 and DEV2.
- Create ticket without assigneeId.
- Expected:
  older registered developer selected.

Auto-assignment audit expectation (all cases where assignment actually happened):
  audit_logs contains action=AUTO_ASSIGN, actor_type=SYSTEM, entity_type=TICKET.

10) Negative and edge-case tests
========================================================================

NE1 - dueDate future:
- Create ticket with future dueDate
- Wait one cycle
- Expect no change.

NE2 - dueDate past:
- Create ticket with past dueDate + LOW/MEDIUM/HIGH
- Expect one-step escalation per cycle.

NE3 - dueDate null:
- Create ticket without dueDate
- Expect no scheduler change.

NE4 - DONE ticket:
- Create or update ticket to DONE with past dueDate
- Expect no scheduler change.

NE5 - Soft-deleted ticket:
- Create past-due ticket then delete it
- Expect no scheduler change.

NE6 - CRITICAL overdue ticket:
- Create CRITICAL past-due
- Expect CRITICAL unchanged and isOverdue=true after cycle.

NE7 - Invalid enum values:
- Try POST /tickets with invalid priority/status/type
  example:
    "priority": "URGENT"
- Expect 400 validation/binding error.

NE8 - Missing required fields:
- Omit title/status/priority/type/projectId in POST /tickets
- Expect 400.

NE9 - Unauthorized setup endpoint access:
- Call POST /tickets without token
- Expect 401.

NE10 - Scheduler must not change status:
- For escalated ticket, compare status before/after cycle.
- Expect status unchanged.

11) Manual timing instructions
========================================================================

Where interval is configured:
- src/main/resources/application.yaml
  issueflow.ticket.escalation.fixed-delay-ms: 60000

How long to wait:
- Wait 70-90 seconds after creating test data for one cycle.
- For multi-cycle tests, wait same interval between checks.

How to identify scheduler run:
- There is no explicit scheduler log line in current scheduler code.
- Practical confirmation methods:
  1) Ticket fields changed in DB/GET response.
  2) New AUTO_ESCALATE rows in audit_logs.

Can interval be reduced locally?
- Yes, for local testing only, you can reduce fixed-delay-ms
  (example: 5000).
- This is safe for local manual testing, but do not use tiny intervals in
  production-like settings.
- If changed locally for convenience, restore or document your local-only value.
