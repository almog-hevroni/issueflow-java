# IssueFlow Swagger Manual Testing Guide (Full)

This document is based only on the currently implemented code in this repository.
It is intended for manual end-to-end testing through Swagger UI, including:
- happy paths
- edge/negative cases
- authentication/authorization behavior
- audit and DB verification

---

## 2) Main Testing Order / Flow

Follow this order to avoid missing prerequisites:

1. Login as seeded admin (`POST /auth/login`).
2. Verify token works (`GET /auth/me`).
3. Create users (ADMIN + DEVELOPER test users) with `POST /users`.
4. Create project(s) with `POST /projects`.
5. Ensure project-linked developers exist (important note):
   - There is no API endpoint to add project members.
   - A DEVELOPER project owner is auto-linked on project creation.
   - For multi-developer workload/auto-assignment tests, add rows to `project_members` via SQL (section 17).
6. Create tickets (`POST /tickets`) with mixed statuses/priorities/assignees/dueDates.
7. Run ticket retrieval and update transition tests.
8. Run comments + mentions tests.
9. Run dependency tests.
10. Run attachment tests.
11. Run CSV export/import tests.
12. Run soft delete/restore tests for tickets/projects.
13. Run workload and auto-assignment tests.
14. Run audit log filter tests.
15. Run scheduler (auto-escalation) tests (requires waiting for scheduler interval).
16. Run logout + token revocation + authz negative tests.
17. Optional SQL verification for all major tables.
18. Final checklist.

## 5) Authentication Endpoints

### 5.1 POST /auth/login
- Purpose: obtain JWT token.
- Auth required: No.
- Headers: `Content-Type: application/json`

Happy path body:
{
  "username": "admin",
  "password": "Password123!"
}

Expected: `200`
Important fields:
- `accessToken`
- `tokenType` = `Bearer`
- `expiresIn` (default 3600)

Copy:
- `accessToken` -> <ACCESS_TOKEN>

Negative body (wrong password):
{
  "username": "admin",
  "password": "WrongPassword!"
}
Expected: `401`

Negative body (missing username):
{
  "password": "Password123!"
}
Expected: `400`

### 5.2 POST /auth/logout
- Purpose: revoke current token.
- Auth required: Yes.
- Headers: `Authorization: Bearer <ACCESS_TOKEN>`
- Body: none
- Expected: `200`

Negative:
- Missing Authorization header -> `401` (security layer)
- Malformed Authorization header on logout (for example `Token abc`) -> `400` with message like `Authorization header must be Bearer token`

### 5.3 GET /auth/me
- Purpose: verify authenticated principal.
- Auth required: Yes.
- Headers: `Authorization: Bearer <ACCESS_TOKEN>`
- Expected: `200`
- Important fields: `id`, `username`, `email`, `fullName`, `role`

Copy:
- Admin `id` -> <ADMIN_ID>

Negative:
- No token -> `401`
- Malformed token -> `401`
- Revoked token (after logout) -> `401`

## 8) Ticket Endpoints

### 8.1 GET /tickets?projectId={projectId}
- Auth required: Yes
- Expected: `200`
- Negative invalid/missing project -> `404`/`400` depending input

### 8.2 GET /tickets/{ticketId}
- Auth required: Yes
- Expected: `200` or `404`

### 8.3 POST /tickets
- Auth required: Yes

Body example with explicit assignee:
{
  "title": "API Ticket Explicit",
  "description": "Assigned explicitly",
  "status": "TODO",
  "priority": "LOW",
  "type": "BUG",
  "projectId": <PROJECT_ID>,
  "assigneeId": <DEV1_ID>,
  "dueDate": "2030-01-01T10:00:00Z"
}

Body example without assignee (auto-assignment path):
{
  "title": "API Ticket Auto",
  "description": "No assignee field",
  "status": "TODO",
  "priority": "MEDIUM",
  "type": "FEATURE",
  "projectId": <PROJECT_ID>
}

Expected: `200`
Important fields:
- `id`
- `assigneeId` (may be null if no linked DEVELOPER exists)
- `isOverdue` initially false

Negative:
- assignee is ADMIN -> `400`
- assignee DEVELOPER not linked to project -> `400`
- project not found -> `404`

### 8.4 PATCH /tickets/{ticketId}
- Auth required: Yes
- Body can include any subset of:
{
  "title": "New title",
  "description": "New description",
  "status": "IN_PROGRESS",
  "priority": "HIGH",
  "assigneeId": <DEV2_ID>,
  "dueDate": "2030-01-03T00:00:00Z"
}

Expected: `200`

Status transition rules to test:
1. TODO -> IN_PROGRESS (valid, `200`)
2. IN_PROGRESS -> IN_REVIEW (valid, `200`)
3. IN_REVIEW -> DONE (valid if dependencies resolved, `200`)
4. TODO -> DONE (invalid skip, `400`)
5. IN_REVIEW -> TODO (invalid backward, `400`)
6. DONE ticket any update (immutable) -> `400`

Other negatives:
- empty body `{}` -> `400`
- assignee ADMIN -> `400`
- assignee not linked DEVELOPER -> `400`

### 8.5 DELETE /tickets/{ticketId}
- Auth required: Yes
- Expected: `200` (soft delete)
- After delete, `GET /tickets/{ticketId}` -> `404`

### 8.6 GET /tickets/deleted?projectId={projectId} (ADMIN only)
- Auth required: Yes
- ADMIN: `200`
- DEVELOPER: `403`

### 8.7 POST /tickets/{ticketId}/restore (ADMIN only)
- Auth required: Yes
- ADMIN: `200`
- DEVELOPER: `403`

Negative:
- restore non-deleted ticket -> `400`

## 11) Attachment Endpoints

## 12) CSV Endpoints

### 12.1 GET /tickets/export?projectId={projectId}
- Auth required: Yes
- Expected: `200`
- Response content-type: `text/csv`
- Verify header fields:
  id,title,description,status,priority,type,assigneeId
- Verify values are present and properly quoted for commas/quotes.

### 12.2 POST /tickets/import (multipart/form-data)
- Auth required: Yes
- Form fields:
  - `file` (CSV)
  - `projectId` (Long)

Expected response shape:
{
  "created": <number>,
  "failed": <number>,
  "errors": [ "...row error..." ]
}

Happy file:
id,title,description,status,priority,type,assigneeId
1,Imported A,Desc A,TODO,LOW,BUG,<DEV1_ID>
2,Imported B,Desc B,IN_PROGRESS,MEDIUM,FEATURE,<DEV2_ID>

Expected: created increments, failed 0.

Quotes/commas file:
id,title,description,status,priority,type,assigneeId
1,"Imported, Ticket","quoted ""text""",TODO,LOW,BUG,<DEV1_ID>

Expected: `200`, successful parse.

Invalid enum file:
id,title,description,status,priority,type,assigneeId
1,Bad Enum,Desc,NOT_A_STATUS,LOW,BUG,<DEV1_ID>

Expected: `200` overall, row counted as failed, error entry includes invalid status.

Invalid assignee file:
id,title,description,status,priority,type,assigneeId
1,Bad Assignee,Desc,TODO,LOW,BUG,999999

Expected: row failure with not found/invalid assignee message.

Important audit behavior:
- Audit CREATE log is written only for each successfully imported row.
- Failed rows do not create CREATE audit for ticket.

## 13) Audit Log Endpoint

### 13.1 GET /audit-logs
- Auth required: Yes
- Query params (optional):
  - entityType (String)
  - entityId (Long)
  - action (enum):
    CREATE, UPDATE, DELETE, RESTORE, AUTO_ASSIGN, AUTO_ESCALATE, LOGIN, LOGOUT
  - actor (enum):
    USER, SYSTEM

Base call:
- `GET /audit-logs` -> `200` list

Filter examples:
- `/audit-logs?entityType=TICKET`
- `/audit-logs?entityType=TICKET&entityId=<TICKET_ID>`
- `/audit-logs?action=CREATE`
- `/audit-logs?actor=SYSTEM`
- combined:
  `/audit-logs?entityType=TICKET&entityId=<TICKET_ID>&action=CREATE&actor=USER`

Verify audit appears after:
- user/project/ticket/comment create/update/delete
- ticket/project restore
- auth login/logout
- auto-assignment (SYSTEM + AUTO_ASSIGN)
- auto-escalation (SYSTEM + AUTO_ESCALATE)
- csv import successful rows (source tickets-csv-import in detailsJson)

## 14) Security Tests (Detailed)

### 14.1 Valid login
- `POST /auth/login` with correct credentials -> `200`

### 14.2 Invalid password
- `POST /auth/login` wrong password -> `401`

### 14.3 Protected endpoint without token
- `GET /users` without Authorization -> `401`

### 14.4 Protected endpoint with malformed token
- `GET /users` with `Authorization: Bearer not-a-jwt` -> `401`

### 14.5 Revoked token after logout
1. login -> get token
2. call `POST /auth/logout` with token -> `200`
3. call `GET /auth/me` with same token -> `401`

### 14.6 ADMIN-only endpoint with DEVELOPER token
1. login as DEVELOPER -> <ACCESS_TOKEN_DEV>
2. call `GET /projects/deleted` -> `403`
3. call `GET /tickets/deleted?projectId=<PROJECT_ID>` -> `403`
4. call restore endpoints as developer -> `403`

### 14.7 Verify structured errors
For 401/403/400 endpoints, verify JSON fields:
- timestamp
- status
- error
- message
- path

## 18) Workload / Auto-Assignment Tests

Prerequisite:
- Multiple DEVELOPER users linked to same project in `project_members`.
- Because no API exists for linking members, insert rows via SQL.

### 18.1 Workload scope tests
1. `GET /projects/{projectId}/workload`
2. Verify only linked DEVELOPER users are returned.
3. Verify ADMIN users are not returned.
4. Verify DEVELOPER users not linked to this project are not returned.

### 18.2 Auto-assignment least loaded
1. Ensure DEV1 has more open tickets than DEV2.
2. Create new ticket without `assigneeId`.
3. Verify assigned to least-loaded linked DEVELOPER.

### 18.3 Tie-break rule
If linked developers have same open count, assignment should prefer oldest registered developer (`created_at` earlier).

### 18.4 No candidates
If project has zero linked DEVELOPER members, create ticket without `assigneeId` and verify response has:
- `assigneeId: null`

## 23) Auto-Escalation Scheduler Tests

Scheduler is not directly triggerable via Swagger. It runs automatically every 60 seconds by default.

### 23.1 Setup
Create multiple tickets with past dueDate in non-DONE statuses:
- LOW
- MEDIUM
- HIGH
- CRITICAL

Also create:
- future dueDate ticket
- no dueDate ticket
- DONE ticket with past dueDate

Example body:
{
  "title": "Escalation Candidate LOW",
  "description": "scheduler test",
  "status": "IN_PROGRESS",
  "priority": "LOW",
  "type": "BUG",
  "projectId": <PROJECT_ID>,
  "assigneeId": <DEV1_ID>,
  "dueDate": "2020-01-01T00:00:00Z"
}

### 23.2 Execution steps
1. Create tickets.
2. Wait at least one scheduler cycle (~60s).
3. Re-fetch each ticket via `GET /tickets/{ticketId}`.
4. Verify priority promotion:
   - LOW -> MEDIUM
   - MEDIUM -> HIGH
   - HIGH -> CRITICAL
   - CRITICAL stays CRITICAL
5. Verify `isOverdue` behavior:
   - escalated priorities keep/return `isOverdue=false` until CRITICAL ticket gets another cycle where overdue toggles true
   - CRITICAL overdue candidate eventually `isOverdue=true`
6. Verify future dueDate ticket does not change.
7. Verify ticket without dueDate does not change.
8. Verify DONE ticket does not change.
9. Verify audit log created with:
   - action = AUTO_ESCALATE
   - actor = SYSTEM
